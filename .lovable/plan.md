
## Что делаю

### 1. Раздел «Партнёры и сотрудничество»

**База данных** (миграция):
- `partner_categories` — `slug`, `title`, `sort_order`
- `partners` — название, категория, описание, преимущества (jsonb), телефон, email, сайт, соцсети (jsonb), логотип, главное фото, галерея (jsonb), активен, порядок
- `partner_applications` — имя, компания, телефон, email, сайт, категория, комментарий, статус (`new`/`in_progress`/`done`), дата
- `partners_content` — редактируемые тексты страницы (hero-заголовок, hero-текст, текст блока «Стать партнёром» и т.д.)
- RLS: публичное чтение партнёров/категорий/контента; админ — запись; заявки — публичная вставка, чтение только админу
- Заполнение 9 базовых категорий и текстов hero/CTA

**Публичная часть:**
- Новая страница `/partners` — hero, фильтр по категориям, сетка карточек, форма «Стать партнёром»
- Страница карточки `/partners/$id` — фото, логотип, описание, преимущества, контакты, соцсети, галерея
- Ссылка «Партнёры» в шапке сайта
- На странице «Квартира под ключ» (`turnkey`) — блок «Проверенные партнёры» с CTA на `/partners`

**Админка:**
- `/admin/partners` — список с переключением активности, перетаскиванием порядка
- `/admin/partners/$id` — форма редактирования партнёра (загрузка логотипа, фото, галереи через существующий прокси `/api/public/photo`)
- `/admin/partners/categories` — CRUD категорий
- `/admin/partners/content` — редактирование hero-текстов раздела
- `/admin/partner-applications` — список заявок со сменой статуса
- Пункты в навигации админки

### 2. Редизайн страницы «О компании»

Визуальная переделка существующей `/about` без потери редактируемости:
- Hero на всю ширину: крупное фото + крупный заголовок + цифры компании поверх
- Сетка преимуществ — иконки + короткие тексты (Lucide-иконки по полю в БД)
- Цифры компании — большие числа в bento-сетке
- «Почему у нас дешевле» / «Почему выбирают нас» — карточки 2-в-ряд с иконками
- Временная шкала «Как мы работаем» (4 шага) — горизонтальная на десктопе, вертикальная на мобильном
- Блок «Мебель у наших покупателей» — крупная масонри-галерея (`customer_photos`) на всю ширину с lightbox
- Галерея производства/шоурума — слайдер с hover-zoom
- FAQ — Accordion (как сейчас)
- Шоурум + кнопка «Построить маршрут»
- CTA-форма консультации
- Анимации: `animate-fade-in` при скролле через IntersectionObserver, `hover-scale` на карточках/фото
- Воздух: большие отступы между секциями (py-20/24), max-w ограничения, мягкие тени

Источник данных не меняется — те же таблицы `about_*`, `customer_photos`, `gallery_items`, `faqs`. Все тексты и фото по-прежнему редактируются через `/admin/about` и связанные разделы.

## Технические детали

- Миграция создаст 4 таблицы с GRANT + RLS (public read для каталога/контента, admin write через `has_role`, заявки — public insert + admin read/update)
- Сидинг категорий и текстов hero — отдельный insert после миграции
- Загрузка изображений — через существующий `uploadPhoto.ts` + прокси `/api/public/photo/$`
- Запросы — TanStack Query в loader-ах публичных страниц через серверные функции с `supabaseAdmin` (публичные read-only)
- Админ-страницы — под `_authenticated/` с проверкой роли admin
- Форма заявки — клиентская валидация через zod, server-fn для вставки в `partner_applications`
- Анимации «О компании» — небольшой хук `useReveal` (IntersectionObserver) + классы из `tailwind.config` (`animate-fade-in`, `hover-scale`)
- Маршруты для дополнительных recommended-партнёров на товаре/в категории закладываю в виде свободного поля `recommended_for` (массив slug категорий товаров) у партнёра — но виджет на странице товара пока не подключаю, чтобы не раздувать скоуп (можно включить отдельным запросом)

## Что НЕ делаю в этой итерации

- Не вывожу рекомендованных партнёров на странице товара/в карточке товара (упомянуто как «предусмотреть на будущее») — поле в БД заложу, виджет подключу следующим шагом по запросу
- Не меняю существующую логику тканей, корзины, заказов
